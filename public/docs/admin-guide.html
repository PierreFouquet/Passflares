1. Organisations Overview

Organisations allow a team to share vaults. Any registered user can create an organisation and becomes its first admin. Admins can invite members and create organisation-owned vaults.

Vault access is controlled per-vault with three permission levels:

  • read — can view and copy entries.
  • write — can add, edit, and delete entries.
  • manage — full control including deleting the vault and managing access.

2. Creating an Organisation

  1. In the vault creation form, set Owner to Organisation.
  2. In the organisation dropdown, choose Create New Organisation…
  3. Enter a name and optional description in the modal that appears.
  4. Click Create Organisation.

You are automatically added as an admin of the new organisation.

3. Adding Members to an Organisation

Only organisation admins can add members.
  1. From the vault list, find a vault owned by the organisation.
  2. Click Manage Org on that vault.
  3. In the modal, enter the member's email address. They must already have a Passflares account.
  4. Select their role: member or admin.
  5. Click Add Member.

Members gain access to all vaults assigned to the organisation according to the vault's permission level.

4. Creating an Organisation-Owned Vault

  1. In the vault creation form, set Owner to Organisation.
  2. Select the organisation from the dropdown.
  3. Enter a vault name and optional description.
  4. Click Create Vault.
You must be an admin of the organisation to create a vault on its behalf.

The vault is created with manage access for the organisation. All organisation members can access it at their respective permission level.

6. Security Best Practices

For admins

  • Use a different master password for Passflares than any other service.
  • Grant the minimum permission needed — most team members only need read or write, not manage.
  • Regularly review who has access to shared vaults.
  • Remove members promptly when they leave the organisation.
  • Use separate vaults for different sensitivity levels (e.g. one for shared team credentials, one for admin-only credentials).

For all users

  • Never share your master password with anyone, including other admins.
  • Log out when using a shared computer — or let the inactivity timer handle it.
  • Export your data regularly as a backup.
  • Report suspicious activity (unexpected logins, entries you didn't create) immediately.

7. Audit Logs

Every sensitive action is recorded in the audit log, including:

  • Successful and failed login attempts (with IP address)
  • Account registration
  • Vault creation, access, data upload, and deletion
  • Organisation creation and member additions
  • Master password changes

Audit logs are accessible to Passflares administrators via the Cloudflare D1 database. Query example:

SELECT * FROM audit_logs ORDER BY timestamp DESC LIMIT 100;

8. Account Management

Deleting an account

Users can delete their own account from the app settings. This permanently removes:

  • All personal vaults and their encrypted data from R2
  • Their user record and credentials from D1
  • Their organisation memberships
  • All audit log entries linked to their account
Deleting an account does not delete organisation-owned vaults. If the user was the sole admin of an organisation, that organisation's vaults will still exist but may become unmanageable. Assign another admin before deleting your account.

9. Infrastructure Reference

Passflares runs on the following Cloudflare services:

ServicePurpose
WorkersAPI routing and request handling
D1 (SQLite)User accounts, vault metadata, organisations, audit logs
R2Encrypted vault blobs (the actual password data)
KVRate limiting counters for login attempts
TurnstileBot protection on auth forms
AssetsStatic frontend files (HTML, CSS, JS)

Secrets

Two secrets must be set in Cloudflare Workers (via wrangler secret put):

  • JWT_SECRET — signs and verifies auth tokens. Must be at least 32 characters, randomly generated.
  • TURNSTILE_KEY — the Turnstile secret key from the Cloudflare dashboard.

Database migrations

To initialise or reset the database schema:

wrangler d1 execute secure-password-db \
  --file=./migrations/0001_init.sql \
  --remote
